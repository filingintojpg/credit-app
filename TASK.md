## Часть 1. Разработка основного функционала сервисов

### Сервис 1

Данный сервис является входной точкой для клиента.

```POST /application/create``` - метод создания заявки. На вход в JSON-формате приходит заявка на кредит от клиента. На выход необходимо отдать идентификатор заявки.

Данные которые должен принимать на вход сервис:
* Фамилия, имя, отчество
* Паспортные данные
* Семейное положение
* Адрес прописки
* Контактный номер телефона
* Сведения о занятости (период работы, должность, название организации)
* Сумма, срок желаемого кредита

Полученные данные необходимо сохранить в БД. Далее создать по API Camunda бизнес процесс с передачей параметра идентификатора заявки, то есть отправить заявку в Camunda (Сервис 2) для принятия по ней решения.

```POST /application``` - метод получения пагинацией данных заявок. Данный метод из БД по фильтру вытаскивает все данные по заявке с учетом текущего решения принятого Сервисом 2.

Фильтры:
* Список статусов заявки
* Сумма
* Срок
* Номер телефона

```GET /application/${applicationId}/status``` - получения статуса принятия решения по определенной заявке.

### Сервис 2

Данный сервис реализует логику обработки кредитной заявки от клиента и принятия по ней решения. Все делегаты реализуются в нем.

Необходимо использовать архитектуру встраивания Camunda (https://docs.camunda.org/manual/7.19/introduction/architecture/) - Embedded Process Engine. 

Необходимо реализовать бизнес процесс принятия решения по заявке. На вход бизнес процессу подается id заявки, созданной в Сервисе 1. 
Бизнес процесс на основе следующих правил принимает решение о выдаче кредита:
* Сумма должна быть в диапазоне от 1 000 до 30 000
* Срок от 3 дней до 30 дней
* Возраст от 18 до 60
Если условия не выполняются необходимо отказать по заявке, иначе одобрить. Факт принятого решения бизнес процесса необходимо сохранить в отдельную таблицу.

## Часть 2. Обработка зависших заявок
Необходимо реализовать новый бизнес процесс, который будет по таймеру (например, раз в 10 минут) брать заявки, которые не успели обработаться по каким-то причинам и в цикле брать в работу. 
Обработку цикла необходимо делать параллельно средствами нотации BPMN + Camunda. Логику обработки с части 1 не нужно дублировать, а важно переиспользовать уже существующий процесс, то есть чтобы для процесса обработки по таймеру и из части 1 был единый бизнес процесс.

## Часть 3. Кэширование данных заявки
Необходимо для сервисов 1 и 2 подключить общее кэширование Redis для сущности заявки. При создании, обновлении заявки сохранять данные в кэш. 
Все чтения для полученных данных одной заявки должны сначала происходить из Redis, если там данные отсутствуют, то загружать из БД. 
То есть реализовать паттерн Read-Through (https://system-design.space/chapter/caching-strategies/).
TTL хранения данных в кэше - 10 минут с момента последнего создания/изменения.
